🌿 What this site is for

This is a self-contained manual test bed for Lucky Orange's browser tracking script (lo.js). It covers a wide range of DOM patterns, input types, and third-party embedding scenarios that LO needs to handle correctly during session recording.

Use it to:

  • Verify recording behavior after changes to fingerprint or core
  • Test lo-sensitive scrambling and iframe placeholder logic
  • Check shadow DOM traversal with open vs. closed roots
  • Reproduce edge cases before filing or closing a bug
  • Compare behavior between production and staging tracking scripts side-by-side (different browser tabs)
💡 This site does not have a backend. Everything runs in the browser. There is nothing to install beyond a static file server.

🚀 Running it locally

Serve the test-sites/ directory with any static file server. A serve.json is included that sets the CORS and Private Network Access headers needed to load lo.js from a local dev server.

Option A — npx serve (recommended)

# from the test-sites directory
npx serve .

This uses the included serve.json automatically. The site will be available at http://localhost:3000 (or the next available port).

Option B — any other server

# python
python3 -m http.server 3000

# or VS Code Live Server extension — just open index.html
⚠️ If you're testing against a local core dev server (e.g. https://localhost:8080), the test site must also be served from localhost — not from audryf.github.io. Chrome's Private Network Access policy blocks public HTTPS pages from loading scripts from loopback addresses. Serving both from localhost sidesteps this entirely.

Why serve.json exists

The serve.json in this directory adds these response headers to every file served:

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
Access-Control-Allow-Headers: X-Requested-With, content-type, Authorization
Access-Control-Allow-Private-Network: true

The last header (Access-Control-Allow-Private-Network) is specifically what Chrome requires when a page makes requests to loopback addresses from a non-loopback origin.

⚙️ Switching LO environment & site ID

The dark bar pinned to the very top of the page is the LO Config Bar. It lets you change which tracking script is loaded without touching any code.

Prod / Staging toggle

Switches between the production CDN and the staging GCS bucket. Selection is saved to localStorage.

Site ID input

Override the site ID loaded with the script. Leave blank to use the default for the selected environment.

Apply & Reload

Saves the config and reloads the page. On reload, the inline script in <head> reads localStorage and injects the right script tag before anything else loads.

How the loader works

The very first <script> tag in <head> is an inline IIFE that runs synchronously before any other resource loads:

// Reads config, builds the correct src, and injects a script tag
var cfg = JSON.parse(localStorage.getItem('lo-config') || '{}')
var env = cfg.env || 'production'
var siteId = cfg.siteId || DEFAULT_IDS[env]
var s = document.createElement('script')
s.async = s.defer = true
s.src = URLS[env] + '?site-id=' + siteId
document.head.appendChild(s)

This means the tracking script is loaded with the same timing as a hard-coded <script async defer> tag — there is no difference in LO's initialization behavior.

Simulated third-party widgets

Two persistent widgets live outside the main sections and are always present on the page:

  • Chat widget (bottom-right corner) — simulates an Intercom/Drift-style chat button. The window opens, accepts typed messages, and sends a fake reply. Tests LO recording inside a floating overlay that is always in the DOM.
  • Cookie consent banner (bottom of page) — simulates a OneTrust/Cookiebot-style banner. Dismiss it with Accept or Reject. Tests LO tracking on a fixed-position overlay that removes itself from the DOM on interaction.

🔐 lo-sensitive class reference

Any element (or its ancestor) with one of the following class names is treated as sensitive by LO's scrambleNode function in fingerprint/src/tree-mirror/privacy.js:

Class nameNotes
lo-sensitivePrimary class, recommended
lo_sensitiveUnderscore variant
losensitiveNo-separator variant
loscrambleAlias
lo_scrambleAlias with underscore

Sensitivity inherits downward — you can put the class on a container and all children will be treated as sensitive. You can opt a child back out with lo-not-sensitive or lonotsensitive.

What scrambleNode does per element type

ElementBehavior in recording
Text nodesCharacters replaced with random letters/numbers preserving case and format
<input>, <textarea>value attribute scrambled; interaction events still fire
<img>src replaced with a 1×1 transparent GIF (BLANK_IMG) — image appears blank
<iframe>src set to about:blank, srcdoc injected with a styled placeholder screen — content never loads in the player
Any element with titletitle attribute scrambled
📌 The iframe placeholder (srcdoc) was added to privacy.js to replace the previous behavior where a sensitive iframe showed as an empty white box. The player now renders a "Third-party iframe — content not captured" message instead.

🧩 Shadow DOM & web components

The Web Components section contains four custom elements that cover the main shadow DOM scenarios LO needs to handle:

ElementShadow modeExpected LO behavior
<plant-card> open LO can access shadowRoot and record the internal DOM. Content should appear in recordings.
<secret-note> closed LO cannot access shadowRoot (returns null). The element appears in recordings as an empty custom element.
<mood-badge> None (light DOM) Rendered directly into the main document. Recorded normally like any other element.
<star-rating> open Interactive — click events inside the shadow root should be captured. Star selection re-renders the shadow DOM, which LO should record as mutations.
💡 LO traverses open shadow roots using shadowRoot.childNodes in the initial tree snapshot and subscribes to mutations within them. To verify this is working, open a recording and confirm that the <plant-card> internals (name, description, mood tag) are visible in the DOM panel.

🖼 iframe recording behavior

There are four iframes in the iFrames section, covering the main combinations of cross-origin + sensitivity:

iframeCross-originlo-sensitiveExpected in player
Wikipedia (Houseplant) ✅ Yes ❌ No src neutralized to about:blank by frames.js — shows blank frame
YouTube ✅ Yes ❌ No Same — src neutralized, blank frame
OpenStreetMap ✅ Yes ✅ Yes Placeholder screen: "Third-party iframe — content not captured"
Wikipedia (Lucky Orange) ✅ Yes ✅ Yes Placeholder screen: "Third-party iframe — content not captured"

Where each behavior is implemented

  • Cross-origin src neutralization — core/src/modules/page-recording/transformations/frames.js — detects cross-origin iframes on every vnode pass and sets vnode.a.src = 'about:blank'.
  • lo-sensitive placeholder — fingerprint/src/tree-mirror/privacy.js in scrambleNode() — sets vnode.a.src = 'about:blank' and injects vnode.a.srcdoc with a styled placeholder HTML document. Because srcdoc takes precedence over src in the browser, the player renders the placeholder instead of a blank frame.
📌 There is also a sensitive iframe in the Sensitive Data section (Wikipedia / Lucky Orange page) — that one is the primary test case for the srcdoc placeholder behavior since it has lo-sensitive directly on the wrapper.

➕ Adding new test cases

The site is a single index.html file. Each test is a <section id="..."> with a matching <a href="#..."> in the nav.

Adding a new section

  1. Add a <li><a href="#your-id">Label</a></li> to the .nav-links list.
  2. Add a <section id="your-id" class="section"> block in the body where you want it to appear.
  3. Give it a <h2 class="section-title"> and a <p class="section-desc"> describing what it tests.

Conventions

  • Keep the section-desc short and specific — it's the first thing another developer reads to understand what's being tested.
  • If you're testing a specific LO behavior, mention the file/function it exercises (e.g. "tests scrambleNode in privacy.js").
  • For sensitive data tests, always wrap in a .sensitive-box div with a .sensitive-tag label so the intent is visually obvious.
  • For iframe tests, use the .iframe-card + .iframe-label pattern so it's clear what each iframe is and why it's there.

Third-party scripts

You can add additional third-party embed scripts (Typeform, Calendly, Stripe Elements, etc.) directly to index.html. Just add a new section and drop in the embed code. If the embed is sensitive, wrap it in <div class="lo-sensitive">.

⚠️ Be careful about adding third-party scripts that set their own cookies or make network requests — those could interfere with what you're measuring in LO recordings. Prefer iframe-based embeds over injected scripts when possible for test isolation.
